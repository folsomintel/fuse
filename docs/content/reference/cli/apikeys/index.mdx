`fuse apikeys` (aliases `apikey`, `keys`) manages revocable API keys.
Orchestrator-global and all-or-nothing, these commands require the master token
(`ORCH_AUTH_TOKEN`); the server rejects an API-key-authenticated caller with
403. The `--master` marker on a context is informational client-side only and
gates nothing, a context connected without it but holding the real master token
works. See [Authentication](/reference/auth) for the full token model.

:::warning[Known limitation]
API keys are Postgres-backed. Against an orchestrator started without
`DATABASE_URL` the `/v1/api-keys` routes are not mounted at all, so these
commands fail with the server's message followed by the suffix
`(is this the orchestrator? the url may be wrong, or point at a different service)`,
which points at the wrong problem entirely. The suffix is appended to whatever
the server returned, so match on it rather than on the whole line.
:::

## Commands

| Command | Purpose |
| --- | --- |
| [`fuse apikeys create`](/reference/cli/apikeys/create) | Issue a new API key. |
| [`fuse apikeys list`](/reference/cli/apikeys/list) | List key metadata. |
| [`fuse apikeys revoke`](/reference/cli/apikeys/revoke) | Revoke a key. |
