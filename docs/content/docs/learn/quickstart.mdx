This walks through the fastest path to a running environment. It uses the
orchestrator's built-in in-memory stub provider, so you don't need a real
Firecracker host yet — that's covered in [Cloud](/docs/cloud/firecracker-host-setup)
once you're ready to run real workloads.

## 1. Build and run the orchestrator

```bash
go build -o bin/orchestrator ./cmd/orchestrator
./bin/orchestrator
```

With no `FIRECRACKER_BASE_URL` set, the orchestrator runs against an in-memory stub
provider — handy for trying things out locally. It listens on `:8080` by default.

## 2. Install the CLI

The CLI binary is `fuse`, built from the repo's `cli/` package:

```bash
go build -o bin/fuse ./cli
```

See [CLI overview](/docs/reference/cli) for global flags and config file layout.

## 3. Connect to the orchestrator

```bash
fuse connect http://localhost:8080
```

Since this local orchestrator has no `ORCH_AUTH_TOKEN` set, no `--token` is needed —
that's insecure/dev mode, fine for local exploration only.

`fuse quickstart` walks the same connect step interactively and registers a host on
the way through, which is the path to take once you're moving to a real host rather
than the stub provider.

## 4. Scaffold a Fusefile

```bash
fuse init
```

This writes a fully commented example `Fusefile` to the current directory. Open it
and adjust `resources`, `run`, and any `services` you need. See
[Fusefile](/docs/learn/concepts/fusefile) for the full field reference.

## 5. Bring it up

```bash
fuse up --secret pg_password=devpassword
```

`up` compiles the Fusefile and creates an environment from it, then streams
provisioning events until the environment reaches a terminal state (or pass
`--no-wait` to return immediately).

<Callout type="warn" title="Known limitation">
"Terminal" means `destroyed` or `failed`, and `running` is neither, so a
successful `up` keeps streaming instead of exiting. On a terminal you can press
`q` once the environment reaches `running`. When output is piped or scripted
there is no exit path at all, so pass `--no-wait` and check the result with
`fuse environment get` instead. Tracked as
[issue #50](https://github.com/folsomintel/fuse/issues/50).
</Callout>

The scaffold from step 4 declares `pg_password` under `secrets`, so `up` refuses to
create anything until you supply a value for it, either with `--secret` as above or
with `--secrets-file`. Values live outside the Fusefile by design, the file names the
secrets an environment requires rather than carrying them.

## 6. Check on it

```bash
fuse environment list
fuse environment get <id>
```
