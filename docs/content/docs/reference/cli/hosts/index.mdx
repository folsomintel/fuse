The `fuse hosts`/`fuse host` commands manage the compute hosts the scheduler
places microVMs onto. `fuse hosts` (plural) lists hosts; `fuse host` (singular)
with an ID selects an active host for the current context, and its subcommands
register, inspect, and manage individual hosts.

## Commands

| Command | Purpose |
| --- | --- |
| [`fuse hosts list`](/docs/reference/cli/hosts/list) | List all registered hosts. |
| [`fuse host <id>`](/docs/reference/cli/hosts/list) | Select an active host, or show the current selection with no ID. |
| [`fuse host register`](/docs/reference/cli/hosts/register) | Register a new compute host. |
| [`fuse host get`](/docs/reference/cli/hosts/get) | Show one host's details. |
| [`fuse host cordon`](/docs/reference/cli/hosts/cordon) | Mark a host unschedulable. |
| [`fuse host uncordon`](/docs/reference/cli/hosts/uncordon) | Return a host to active scheduling. |
| [`fuse host remove`](/docs/reference/cli/hosts/remove) | Deregister a host. |
| [`fuse host metrics`](/docs/reference/cli/hosts/metrics) | Show capacity vs. allocated for a host. |

## Active host selection

Many other commands (`fuse environment list`'s default filter, for example)
scope themselves to whichever host is currently selected for the active
context. This is a CLI-local convenience, not an API concept, `fuse environment
create` schedules orchestrator-wide regardless of the selection; use `--region`
on create to influence placement instead. See
[Hosts](/docs/learn/concepts/hosts) for the concept-level model.
