## fuse up / fuse init

The [Fusefile](/docs/learn/concepts/fusefile) path:

```bash
fuse init [path] [-f file] [--force]
fuse up [path] [flags]
```

`up` flags:

| Flag | Purpose |
| --- | --- |
| `-f, --file` | Path to the Fusefile (default: `./Fusefile`, or a positional path). |
| `--secret key=value` | Secret, repeatable. Overrides `--secrets-file` on key collision. |
| `--secrets-file` | Path to a file of `KEY=VALUE` secret lines (`#`-prefixed lines and blanks ignored). |
| `--task-id` | Environment task ID (default: the Fusefile's parent directory name). |
| `--no-wait` | Create the environment without streaming provisioning events. |

## fuse environment

```bash
fuse environment list [flags]      # aliases: environments, env, envs
```

| Flag | Purpose |
| --- | --- |
| `--task-id` | Filter by task ID. |
| `--state` | Filter by state: `provisioning`\|`running`\|`draining`\|`destroying`\|`destroyed`\|`failed`. |
| `--host-id` | Filter by host ID (default: the active host). |
| `--all` | List across all hosts, ignoring the active host. |

```bash
fuse environment get <id>
fuse environment create [flags]
```

`create` flags: `--task-id` (required), `--cpus`, `--ram-mb`, `--storage-gb`,
`--region`, `--max-runtime` (seconds), `--manifest` (inline or `@path`),
`--startup-script` (inline or `@path`), `--gateway-url`, `--gateway-token`,
`--secret key=value` (repeatable), `--follow` (stream provisioning events). With no
`--task-id` in an interactive terminal, required fields are collected via a form.

Note: `create` schedules orchestrator-wide, the active host selection does not pin
placement; use `--region` to influence it.

```bash
fuse environment destroy <id> [--yes]   # aliases: rm, delete
fuse environment drain <id>
fuse environment fork <id> [--reuse-snapshot <id>] [--comment "..."]
fuse environment rotate-token <id>      # alias: rotate
fuse environment watch <id>             # alias: follow
```

`drain` is phase-1 only, it does not auto-destroy; follow up with `destroy` to
remove the environment.

## fuse environment exec

Run a command inside a running environment's guest and print its output:

```bash
fuse environment exec <id> -- ls -l /var/log
fuse environment exec <id> --shell 'ls /var/log | wc -l'
fuse environment exec <id> --timeout 30s -- ./slow-job
```

| Flag | Purpose |
| --- | --- |
| `--shell` | Run a one-liner under `sh -lc`, for pipelines, redirects, and globs. Mutually exclusive with a command after `--`. |
| `--timeout` | Bound the command inside the guest (e.g. `30s`, `5m`). Default: the server default of 60s; the server caps at 10m. |

Argv after `--` is the default: it needs no quoting rules and cannot be turned into
an injection by interpolating a value. `--shell` is the explicit opt-in for what
argv cannot express.

`exec` exits with the guest command's exit code, so it composes with `&&` and
`set -e`. A guest command that ran and failed is not a CLI error: stdout and stderr
are still printed, and only the exit code reports the failure.

With `-o json`, the whole result is printed as one object instead:

```json
{ "exit_code": 1, "stdout": "", "stderr": "ls: /nope: No such file or directory\n" }
```

## fuse environment shell

Attach an interactive terminal to a running environment's guest:

```bash
fuse environment shell <id>          # alias: ssh
fuse environment shell <id> -- top
```

With no command this opens the guest's login shell. A command after `--` runs
instead of the shell, still on a terminal.

This is a relayed pty, not SSH: there is no `scp`, no port forwarding, and no
key-based login. It needs a real terminal and refuses to run when stdin and stdout
are not a tty, so for scripted commands use `exec`. The session exits with the guest
command's exit code.

Both `exec` and `shell` require the [master token](/docs/reference/auth). Exec is
root in the guest, and API keys carry no scopes today, so anything weaker would
silently make every issued key a root shell on every VM in the fleet.
