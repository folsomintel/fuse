| Method | Path | Purpose |
| --- | --- | --- |
| `GET` | `/health` | Liveness probe. |
| `GET` | `/ready` | Readiness probe. |
| `GET` | `/metrics` | Prometheus exposition. |
| `GET` | `/v1/version` | Identify the process and its build version. |

All four are mounted **outside** the auth and CIDR-allowlist middleware
chain, entirely by design: Kubernetes/load-balancer probes and Prometheus
scrapers don't (and shouldn't) carry bearer tokens. `/health` and `/ready` are
formal path entries in `internal/api/openapi.yaml`; `/metrics` is mentioned
only in the spec's prose, not as a path entry; and `/v1/version` does not
appear in the spec at all. This page expands on all four with the metric
inventory the spec doesn't carry.

## Liveness

```
GET /health
```

Always `200`, unconditionally, with no dependency checks:

```json
{ "status": "ok" }
```

Deliberately does no work, kubelet uses this to decide whether to restart the
process. It cannot fail in a way that would trigger a restart loop from a
downstream dependency being unhealthy, that's what `/ready` is for.

## Readiness

```
GET /ready
```

Checks two dependencies. The state store check is bounded by a 2-second
timeout so a wedged database can't hold the probe open indefinitely; the
fleet check is a nil comparison and cannot block:

- **`state_store`**: a bounded `ListVMs` call against Postgres (or the
  in-memory store in dev). Failures here typically mean Postgres is
  unreachable or the connection pool is wedged.
- **`fleet`**: whether the `FleetManager` is initialized (a nil-pointer
  check at boot).

### Response

`200` when both checks pass:

```json
{ "status": "ready", "checks": { "state_store": "ok", "fleet": "ok" } }
```

`503` when either fails, with the specific failure message truncated to 200
characters so a stack trace or driver error can't blow out the probe
response:

```json
{
  "status": "not_ready",
  "checks": {
    "state_store": "dial tcp 10.0.0.1:5432: connect: connection refused",
    "fleet": "ok"
  }
}
```

Use this endpoint, not `/health`, for a load balancer's "should this instance
receive traffic" decision, `/health` says nothing about whether the
orchestrator can actually serve a request end to end.

## Metrics

```
GET /metrics
```

The stock `promhttp.Handler()`, no custom filtering: exposes the orchestrator's
own metrics alongside the default Go runtime and process collectors
(`go_*`, `process_*`).

### Orchestrator-specific metrics

**Reconcile and fleet** (unlabeled):

| Metric | Type | Measures |
| --- | --- | --- |
| `orchestrator_reconcile_duration_seconds` | Histogram | Time spent in one reconcile cycle. |
| `orchestrator_reconcile_cycles_total` | Counter | Total reconcile cycles completed. |
| `orchestrator_fleet_tracked_vms` | Gauge | VMs currently tracked by the fleet manager. |
| `orchestrator_fleet_provider_vms` | Gauge | VMs reported by the provider(s). |
| `orchestrator_reconcile_orphans_destroyed_total` | Counter | Orphaned VMs the reconcile loop destroyed. |
| `orchestrator_reconcile_orphans_failed_total` | Counter | Orphan destroy attempts that failed. |
| `orchestrator_reconcile_orphans_dead_lettered_total` | Counter | Orphans given up on after repeated failures. |
| `orchestrator_reconcile_stuck_tasks_suspected_total` | Counter | Tasks that exceeded the runtime ceiling once (first strike, no action taken yet). |
| `orchestrator_reconcile_stuck_tasks_failed_total` | Counter | Tasks failed and their VMs torn down for exceeding the runtime ceiling twice (second strike). |
| `orchestrator_reconcile_vms_missing_provider_total` | Counter | VMs tracked in the state store with no matching provider record. |

**HTTP (RED metrics)**:

| Metric | Type | Labels | Measures |
| --- | --- | --- | --- |
| `orchestrator_http_requests_total` | CounterVec | `route`, `method`, `code` | Total requests. |
| `orchestrator_http_request_duration_seconds` | HistogramVec | `route`, `method` | Request latency. |
| `orchestrator_http_requests_in_flight` | Gauge | (none) | Concurrent in-flight requests. |

The `route` label is the chi route **pattern** (for example
`/v1/environments/{vmId}`), not the raw request path, deliberately bounded
cardinality so a client hammering random invalid IDs can't blow up the metric
series count.

See [Monitoring](/docs/cloud/monitoring) for a Prometheus scrape config and
guidance on which metrics matter operationally.

## Version

```
GET /v1/version
```

Always `200`, identifying the process and its build:

```json
{ "service": "fuse-orchestrator", "version": "1.4.0" }
```

`version` renders as `dev` when the binary wasn't stamped at build time.
Despite sitting under the `/v1` prefix, this route is unauthenticated, it is
served by the same handler as `/health` and `/ready` on the outer mux, so
`fuse connect` can confirm it is talking to an orchestrator (rather than a
host agent, or nothing at all) before it holds a validated token.
