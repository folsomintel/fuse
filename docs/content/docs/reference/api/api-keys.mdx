| Method | Path | Purpose |
| --- | --- | --- |
| `POST` | `/v1/api-keys` | Create a new key. |
| `GET` | `/v1/api-keys` | List key metadata. |
| `DELETE` | `/v1/api-keys/{id}` | Revoke a key. |

All three require the master token: a request authenticated with a regular
(non-master) API key is rejected with `403`, even though it passed
`BearerAuth`. Key management is deliberately operator-only, a caller holding
an API key must not be able to mint, list, or revoke keys, including its own.
These three routes exist only when the orchestrator is running with
`DATABASE_URL` set, the key store is Postgres-only. Without it the routes
aren't registered at all and any call returns `404` `route_not_found` rather
than `401`. There is no insecure-mode exemption here: a configured key store
is exactly what makes bearer auth non-optional, so every caller on these
routes carries a principal and is judged on whether it is the master.
See [Authentication](/docs/reference/auth) for the full token model.

## Create

```
POST /v1/api-keys
```

### Request body (optional)

```json
{ "label": "ci-runner" }
```

`label` is an optional human-readable name. The body itself is optional, an
empty body creates an unlabeled key.

### Response

`201` with the new key, `key` is the raw secret and appears **only in this
response**, it cannot be recovered afterward:

```json
{
  "id": "ak_3f2a9c1d5e7b04a8",
  "label": "ci-runner",
  "created_at": "2026-01-15T10:00:00Z",
  "key": "fuse_sk_..."
}
```

### Errors

| Status | Code | When |
| --- | --- | --- |
| `400` | `invalid_argument` | Body present but not valid JSON. An empty body is fine, a malformed one is not. |
| `401` | `unauthorized` | No valid bearer token or session cookie at all. |
| `403` | `unauthorized` | Authenticated, but not with the master token. |
| `500` | `internal` | The key store write failed (Postgres unreachable). |

## List

```
GET /v1/api-keys
```

Returns `200` with `{"api_keys": [...]}`, metadata only:

```json
{
  "api_keys": [
    {
      "id": "ak_3f2a9c1d5e7b04a8",
      "label": "ci-runner",
      "created_at": "2026-01-15T10:00:00Z",
      "last_used_at": "2026-01-16T08:00:00Z"
    }
  ]
}
```

`last_used_at` and `revoked_at` are omitted entirely when unset rather than
serialized as `null`, check for key presence, not for a null value.

The raw secret is never included here, only in the create response, and only
once.

## Revoke

```
DELETE /v1/api-keys/{id}
```

`{id}` is the key's `id` field (from create or list), not the secret itself.
Ids carry the `ak_` prefix while secrets carry `fuse_sk_`, so the two are never
confusable. Returns `204` on success, `404` `not_found` if the ID doesn't
exist. Revocation is immediate: any request authenticated with the revoked
key's secret is rejected from that point forward.
