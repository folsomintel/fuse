# Fuse Documentation Remediation Report

## 1. Executive summary

**These docs are well-written and structurally sound, and substantially untrustworthy in the details.**

The prose is careful, the Diátaxis split is real, cross-links are almost all valid (one broken link in the whole tree). Someone clearly cared. But the docs have drifted from the code in ways that are not cosmetic:

- **The two entry points are both broken.** `learn/quickstart.mdx` inverts the binary names, invents a `fuse-cli` command that exists nowhere in the codebase, and then dead-ends at step 5 because `fuse up` aborts on a required secret the page's own `fuse init` scaffolded. A new reader cannot complete the quickstart. Three of the four SDK quickstart "Full examples" (`sdks/{go,python,typescript}/quickstart.mdx`) hang forever and never reach the `destroy()` they are teaching.
- **Six pages warn readers about a data-corruption bug that was fixed**, and deep-link to the troubleshooting section that says it was fixed and the workaround is unnecessary (`host-agent/fc-agent.py:563-567` has the fix).
- **Ten host-registration examples use a port (`9000`) that appears nowhere in the repo.** The real default is `8090` (`host-agent/fc-agent.py:52`), and `reference/cli/contexts/connect.mdx:41` teaches `8090` as a diagnostic.
- **The error-code contract is systematically wrong.** All three SDK pages promise `conflict` for GPU snapshot/fork and for `deregister` with VMs assigned; the server returns `500 internal` in every case because the orchestrator returns bare `fmt.Errorf` with no sentinel. Readers who write `isConflict(err)` get code that never fires.
- **Every CLI transcript containing a timestamp is fabricated.** `hosts/list.mdx`, `snapshot/list.mdx`, `environment/list.mdx`, `apikeys/list.mdx` all show `"3m ago"` — a string no formatter in the CLI produces (`ago()` at `cli/render.go:125` emits bare `3m`; the columns actually use `shortTime`, an absolute timestamp). Several also show plain-space tables where `renderTable` (`cli/render.go:41`) draws a lipgloss rounded border.

Underneath the doc bugs sits a real code problem: **five distinct orchestrator preconditions return HTTP 500 instead of 409**, because the sentinel-error convention documented in `CLAUDE.md` was not followed. The docs describe the intended behavior; the code never shipped it. Several SDK pages are therefore "wrong" only in the sense that they document a contract the server does not honor.

Verdict: **reference/ is the weakest quadrant despite being the most detailed** — precision is its entire job and the transcripts, status codes, and flag semantics are unreliable. `cloud/` is the strongest. `learn/` has the highest blast radius per error because it is where new readers start.

## 2. Findings by severity

### HIGH — a reader following the docs hits an error or is actively misled

#### H1. Six pages warn about a fixed data-corruption bug
**Files:** `reference/api/snapshots.mdx:130`, `reference/cli/snapshot/restore.mdx:24`, `reference/cli/snapshot/index.mdx:11`, `reference/sdks/{go,python,typescript}/snapshots.mdx:36/30/31`
**Wrong:** Each carries `<Callout type="warn" title="Known limitation">` about restore corrupting data on unsynced writes, pointing "for the workaround" at `cloud/troubleshooting.mdx:76-79`, which says the bug is fixed and the workaround is no longer needed.
**Fix:** Delete all six callouts. `host-agent/fc-agent.py:563-567` copies to `.restore-tmp` then `mv`s over the live rootfs so the restored image lands on a fresh inode. Troubleshooting is correct.

#### H2. `bin/fuse` names both binaries; the quickstart invents `fuse-cli`
**Files:** `learn/quickstart.mdx:15,27,35,44,54,64-65`; also `README.md:108`, `cloud/orchestrator-deploy.mdx:152`, `reference/sdks/{go,python,typescript}/advanced.mdx`
**Wrong:** `.goreleaser.yaml:12-14` ships `./cmd/orchestrator` as `orchestrator`; `:60-62` ships `./cli` as `fuse` (`cli/root.go:57` declares `Use: "fuse"`). Six pages build the orchestrator as `bin/fuse`. The quickstart, which builds both, resolves the collision by renaming the CLI to `bin/fuse-cli` — a name that exists in no release artifact, contradicts `reference/cli/index.mdx:13` (`go build -o bin/fuse ./cli`), and appears in no other page. The quickstart also self-contradicts: line 25 says in prose "The CLI binary is `fuse`", line 27 builds `bin/fuse-cli`.
**Fix, in order:** (a) change every `go build -o bin/fuse ./cmd/orchestrator` to `bin/orchestrator` and its invocation to `./bin/orchestrator`; (b) `learn/quickstart.mdx:27` → `go build -o bin/fuse ./cli`; (c) rename the five `fuse-cli` invocations to `./bin/fuse`. Applying (b)/(c) without (a) makes step 2 overwrite the orchestrator started in step 1.

#### H3. The quickstart dead-ends on its own scaffold
**File:** `learn/quickstart.mdx:51-59`
**Wrong:** `fuse init` scaffolds `secrets:\n  - pg_password` (`cli/init_cmd.go:52-53`); `fuse up` then aborts with `missing required secrets: pg_password` (`cli/up_cmd.go:62-63`) before contacting the orchestrator. Reproduced empirically against a clean stub orchestrator: `init` exits 0, documented `up` exits 1.
**Fix:** Show `fuse up --secret pg_password=devpassword` (verified to reach `running` against the stub), or tell the reader in step 4 to delete **both** the `services` and `secrets` blocks — `services.postgres.env.POSTGRES_PASSWORD` references `{ secret: pg_password }` (`cli/init_cmd.go:35`), so removing `secrets` alone dangles.

#### H4. Ten host-agent URLs use port `9000`, which exists nowhere in the repo

**Wrong:** `host-agent/fc-agent.py:52` defaults `FC_AGENT_PORT=8090`; `cloud/orchestrator-deploy.mdx:38-43` and `cloud/firecracker-host-setup.mdx:69` both say `8090`; `reference/cli/contexts/connect.mdx:41` teaches it as a diagnostic. The GPU examples all correctly use `8091`, which makes `9000` look deliberate.
**Fix:** `9000` → `8090` in all ten.

#### H5. GPU host DNAT range is wrong — a firewall following it breaks at 150 VMs

**Wrong:** `fc-agent.py:56` uses `FUSE_HOST_PORT_BASE=19550`; `qemu-agent.py:78` uses `19650`. Both `pick_index()` over `range(1, 250)`. So Firecracker is `19551-19799` (correct as documented) but GPU hosts are **`19651-19899`**.
**Fix:** Correct `gpu-host-setup.mdx:186` to `19651-19899` and split the deploy port table into two rows.

#### H6. `fuse up <dir>` / `fuse init <dir>` do not append `/Fusefile`
**Files:** `reference/cli/up.mdx:33`, `reference/cli/init.mdx:28-29`
**Wrong:** `resolveFusefilePath` (`cli/up_cmd.go:118-126`) returns `args[0]` verbatim — no stat, no join. `up` then `os.ReadFile`s it; `init` `os.WriteFile`s it. Verified by running the built CLI: `fuse init ./services/api` against a directory fails with "a Fusefile already exists at ./services/api (use --force to overwrite)"; with `--force`, "open ./services/api: is a directory". `fuse up ./services/api` fails `read ./services/api: is a directory`. `learn/use-cases/interactive-debugging.mdx:22` uses the correct file-path form.
**Fix:** `fuse up ./services/api/Fusefile`, `fuse init ./services/api/Fusefile`. Prose at `up.mdx:28-29` is correct and needs no change. (Or fix `resolveFusefilePath` to join — see code bugs.)

#### H7. `fuse environment shell` example passes a task ID
**File:** `learn/use-cases/interactive-debugging.mdx:20-21,25-26`
**Wrong:** VM ids are `prefix + taskID` (`internal/orchestrator/fleet.go:456`; default prefix `fuse-` at `cmd/orchestrator/main.go:108`). No resolution exists anywhere: `cli/exec_cmd.go:118` → `sdks/go/exec.go:202` → `internal/api/exec.go:127` is pure path interpolation, and `ErrVMNotFound` maps to 404. `fuse environment shell repro-bug-4821` 404s.
**Fix:** Use `fuse-repro-bug-4821` (which `fuse up` itself prints, `cli/up_cmd.go:97`), fix the two shell prompts to `root@fuse-repro-bug-4821`, and add one sentence that the environment id is the vm prefix plus the task id.

#### H8. `rotate-token` is sold as leak remediation but does not revoke anything
**Files:** `learn/concepts/environments.mdx:85-92`, `reference/cli/environment/rotate-token.mdx:11-13`
**Wrong:** `fused` reads `--auth-token-file` once at process start (`cmd/fused/main.go:80-86,101`) with no watcher or SIGHUP handler. `RotateToken` (`internal/orchestrator/token_rotation.go:26-78`) uploads new credential files and re-encrypts the orchestrator's copy but never calls `StartAgent` — unlike `fork.go:216`, which does exactly that with the comment explaining why it is required. The old token keeps working. `cli/environment_cmd.go:343` prints an unqualified success message.
**Fix:** Add a `<Callout type="warn">`: rotation updates the orchestrator's copy and the guest's credential files but does not restart `fused`, so the leaked token stays valid in the running guest. Point at fork or destroy/recreate for an actual leak. Remove "useful if you suspect a token has leaked."

#### H9. `environments.mdx` claims a fork onto a full host fails — it silently overcommits
**File:** `learn/concepts/environments.mdx:76-78`
**Wrong:** Fork bypasses the scheduler by design (`internal/orchestrator/fork.go:154-156`) and calls `allocateOnHost` (`fleet_hosts.go:189-223`), which increments `Allocated.*` with no capacity comparison and **no error return** — there is no failure path even in principle. `fits()` has exactly one caller, `scheduler.go:279` inside `Schedule`, which fork never reaches.
**Fix:** Replace only the last sentence: "A fork does not go through the scheduler, so it is charged to the source's host without a capacity check: forking onto a full host succeeds and overcommits it." First two clauses are correct and test-pinned (`fork_credentials_test.go:231`).

#### H10. The agent-sandboxes crash-recovery pattern cannot work for a crash
**File:** `learn/use-cases/agent-sandboxes.mdx:57-61`
**Wrong:** Snapshot create (`snapshots.go:59-61`), restore (`:299-301`), and fork (`fork.go:88-90`, checked before the `ReuseSnapshotID` branch) all require the source **running**. A VM that actually dies is deleted from `fm.vms` (`fleet.go:584/650/686`), so a post-crash fork returns `ErrVMNotFound`. There is no standalone restore-into-a-new-VM path (`snapshots.mdx:34` says so).
**Fix:** State that the pattern recovers from a failed *agent step* (VM fine, work went wrong) — which works, since Fuse does no guest liveness checking — and explicitly that it does not cover loss of the VM or host.

#### H11. `artifacts.mdx` describes a drop-in mechanism that does not exist
**File:** `learn/concepts/artifacts.mdx:44-45`
**Wrong:** No drop-in is ever written. `fc-agent.py:689-692` comments "Write a SELF-CONTAINED systemd unit (not a drop-in)" and `:728` overwrites the baked unit via `cat > /etc/systemd/system/fused.service`. `qemu-agent.py:930-956` is identical. `--manifest`/`--secrets` ride inline in the replacement `ExecStart` (`:701-703`); `--gateway`/`--vm-id`/TLS flags ride in `FUSED_EXTRA_ARGS` via `/etc/default/fused` (`:727`). So none of the four flags travel the documented way, and they split across two mechanisms.
**Fix:** Rewrite as the finding proposes. Also correct the stale "drop-in" language at `host-agent/fused.service:9`, `host-agent/README.md:230`, `host-agent/FUSE.md:38` — that comment is the likely origin.

#### H12. Firecracker host-agent contract: false "all routes" claim + two missing routes
**File:** `cloud/firecracker-host-setup.mdx:93,96-108`
**Wrong:** `fc-agent.py`'s `_route` dispatches five patterns; three are outside `/v1/vm`: `GET /v1/capacity` (`:1217`, actively called by `internal/firecracker/firecracker.go:207` as the scheduler's capacity prober), `GET /` and `GET /healthz` (`:1220`). The table also omits `POST /v1/vm/{id}/fork` (`fc-agent.py:1208`, called from `firecracker.go:137`) and `GET /v1/vm/{id}/attach` (`fc-agent.py:1151`, called from `firecracker.go:307`). A bring-your-own agent built from this table cannot serve `fuse fork` or `fuse attach`, and an operator allowlisting only `/v1/vm` breaks capacity probing.
**Fix:** Reword to "VM routes are under `/v1/vm`"; add rows for `/v1/capacity`, `/healthz`, `/v1/vm/{id}/fork`, and `/v1/vm/{id}/attach` (noting attach hijacks the socket for a `fuse-attach/1` pty stream and departs from the table's JSON contract). Note that auth applies to every route including health — `_auth` (`:1072`) has no path exemption, so an unauthenticated probe gets 401 and reports the host down.

#### H13. `POST /login` does not accept API keys
**File:** `reference/api/auth-endpoints.mdx:30-31`
**Wrong:** `internal/api/auth.go:56` is a single `subtle.ConstantTimeCompare(req.Token, h.AuthToken)`; `h.APIKeys` is never consulted in `login()`. `/login` is registered in a bare group at `handlers.go:146-149` with no `BearerAuth`. An API key gets 401.
**Fix:** "The operator's `ORCH_AUTH_TOKEN`. Unlike `/v1/*` routes, `/login` does not accept revocable API keys — the session cookie it issues carries the master token itself, so browser sessions are always master-privileged."

#### H14. `api-keys.mdx` documents an unreachable state
**File:** `reference/api/api-keys.mdx:17-18`
**Wrong:** The three routes register only when `h.APIKeys != nil` (`handlers.go:203`), which requires `DATABASE_URL` (`main.go:200-221`). Whenever they exist, `keys != nil`, so `BearerAuth`'s insecure pass-through (`middleware.go:86`, requires `expectedToken == "" && keys == nil`) can never apply. Without `DATABASE_URL` the routes 404 `route_not_found` — pinned by `api_keys_test.go:260-274`.

#### H16. `--max-runtime 0` is not unlimited
**File:** `reference/cli/environment/create.mdx:27`
**Wrong:** `reconcile.go:123-125` substitutes `TaskStuckTimeout` when `MaxRuntime <= 0`, defaulting to 2h (`fleet.go:290-291`) and not exposed as a flag or env var. #### H17. `--manifest` must be base64; the doc says raw JSON

#### H18. `destroy` documented as idempotent; it 404s and exits non-zero
**Files:** `reference/cli/environment/destroy.mdx:21-23`, `reference/api/environments.mdx:11,178-180`, `reference/sdks/{go,python,typescript}/quickstart.mdx` + `environments.mdx`
**Wrong:** `fleet.go:1061` deletes the VM from the map on success, so a repeat destroy hits `ErrVMNotFound` (`fleet.go:780-783`) → 404 `not_found` (`errors.go:41-45`) → non-nil error from `RunE` (`cli/environment_cmd.go:258-259`) → non-zero exit. The API page's "returns `404`, not an error" is self-contradictory; the CLI page's "Idempotent: ... returns a not-found error" is too. A script under `set -e` aborts on the cleanup step the docs called safe.
**Fix (CLI page):** Drop "Idempotent" and the scripting rationale: "Destroying an environment that no longer exists returns a not-found error and a non-zero exit code, so scripts that tolerate a missing environment must check first or ignore the exit code."
**Fix (API page):** Delete "not an error."
**Fix (SDK pages + `sdks/typescript/src/environments.ts:126`, `snapshots.ts:62`):** Disambiguate rather than delete: "tears the environment down from any state; the environment ends up gone either way, but a second call rejects with `code === "not_found"` — guard with `isNotFound(err)`."

#### H22. All three SDK "Full example" blocks hang and never call `destroy`
**Files:** `reference/sdks/go/quickstart.mdx:46-54`, `python/quickstart.mdx:21-29`, `typescript/quickstart.mdx:26-30`
**Wrong:** `create` returns only once the environment is running (`ProvisionAndAssign` sets `VMStateRunning` before returning, `fleet.go:~622`). The SSE handler returns early only on a terminal snapshot (`internal/api/sse.go:136-142`), otherwise holds the connection on 15s keepalives. All three SDK iterators exit only on `isTerminalState` = destroyed/failed (`sdks/go/events.go:98`, `python/_services/events.py:28`, `typescript/src/events.ts:73`). The Python stream client is built with `timeout=None` (`_transport.py:39`) and the Go SDK uses a no-timeout stream client, so there is no timeout escape. The loop blocks until the ~2h stuck-task ceiling reaps the VM, then `destroy` throws `not_found`.
**Fix:** Break on `running` (the stream's first snapshot event), or use the `AbortController` pattern already shown at `typescript/quickstart.mdx:48-54`, or drop `events()` from the headline example and show it only in the cancellable snippet.

#### H23. TypeScript/Go `hosts` pages invert the 502 condition
**Files:** `reference/sdks/typescript/hosts.mdx:33-35`, `reference/sdks/go/hosts.mdx:38-40`
**Wrong:** `handlers.go:785` gates the entire probe on `provider.(orchestrator.CapacityProber)`. A non-prober provider skips it and hits `handlers.go:815-820` → **400 `invalid_argument`** (pinned by `TestRegisterHost_NoProviderNoCapacityReturns400`, `handlers_test.go:1041`). The 502 fires only when a probe is attempted and fails. Both shipped providers implement `CapacityProber` (`firecracker.go:225`, `qemu.go:183`), so the documented case is effectively unreachable.
**Fix:** Split into three cases: no probe support + unset value → 400 `invalid_argument`; agent rejected the token → 502 `unauthorized` (`handlers.go:798-802`, fires **even if capacity was declared**, because the capacity probe doubles as an auth probe); other probe failure with capacity undeclared → 502 `internal`. Also state that `token` must be the agent's `FC_AGENT_TOKEN`, not the orchestrator token — the token-rejection path is currently undocumented on all three SDK hosts pages.

#### H24. TypeScript `api-keys.mdx` omits the Postgres requirement
**File:** `reference/sdks/typescript/api-keys.mdx:7-9` (also `go/api-keys.mdx`, `python/api-keys.mdx`, `reference/cli/apikeys/index.mdx`)
**Wrong:** Without `DATABASE_URL` the routes are never mounted, so all three methods throw 404 with code `route_not_found` (`handlers.go:169-172`, `types.go:165`). The TS `ErrorCode` enum has no `RouteNotFound` member at all (`sdks/typescript/src/errors.ts:8-15`), so `isNotFound()` returns false and no exported predicate matches. Aggravated by `sdks/*/advanced.mdx`, which recommends exactly the no-`DATABASE_URL` stub setup and claims "every SDK call behaves exactly as it would against a real deployment."
**Fix:** Add up front: API key management requires `DATABASE_URL`. Without it these calls throw 404 `route_not_found` — note `isNotFound()` does not match it. Add the same exception to all three `advanced.mdx` pages.

## 4. Diátaxis structural assessment

**The split is working.** `learn/` is genuinely tutorial+explanation, `cloud/` is genuinely how-to, `reference/` is genuinely reference. Cross-links are dense and (with one exception) valid. The quadrant boundaries are respected far better than in most projects this size. The problems are content accuracy and a handful of misplacements — not structure.

**What's working:**
- `learn/concepts/` is a clean explanation cluster: `environments`, `snapshots`, `hosts`, `providers`, `artifacts`, `fusefile`. These correctly explain *why* rather than *how*, and the reference pages defer to them appropriately.
- `cloud/` is the healthiest quadrant. `firecracker-host-setup.mdx`, `gpu-host-setup.mdx`, `orchestrator-deploy.mdx`, `security.mdx`, `monitoring.mdx`, `troubleshooting.mdx` are all correctly goal-oriented how-to.
- `reference/cli/` mirrors the command tree one-to-one, which is the right reference shape.

**Misplacements and gaps:**

1. **`learn/quickstart.mdx` is a tutorial that doesn't work end-to-end** (H2, H3). A tutorial's single obligation is that following it exactly produces success. This one cannot. It is also the *only* page in the tree using `fuse-cli`. Highest-priority structural fix.

2. **`learn/use-cases/` is how-to material filed under learn.** `ci-runners.mdx`, `agent-sandboxes.mdx`, `interactive-debugging.mdx`, `gpu-workloads.mdx` are goal-oriented recipes ("how do I run CI on Fuse"), not tutorials or explanation. They sit next to `learn/concepts/`, which is correctly explanatory. This is defensible if the intent is "worked examples that teach," but they are written as recipes and carry recipe-shaped bugs — H10's crash-recovery pattern and M8/M9/M10's CI recipe are all failures of *procedural* accuracy, the how-to failure mode. Either move them to `cloud/` (or a `guides/` sibling) or rewrite them explicitly as illustrative rather than runnable.

3. **`cloud/troubleshooting.mdx` is a diagnostic reference wearing how-to clothing.** It is organized by symptom, which is correct, but it is thin (7 headings) and missing the failure mode operators actually hit most (M19). Troubleshooting is arguably a fifth quadrant; the important thing is that it be **symptom-indexed and complete**. Right now it is symptom-indexed and incomplete, and the one entry it does own — snapshot restore corruption — is contradicted by six reference pages (H1).

4. **`reference/api/index.mdx` is doing explanation inside reference.** Its auth model, error-envelope, and request-correlation prose is genuinely useful and genuinely explanatory — and it is where three of the tree's factual errors live (M20, M21, M22), because explanation in a reference page is not maintained with reference discipline. Consider hoisting the auth/error-model narrative into `reference/auth.mdx` (which already exists and already covers half of it, inconsistently — see M23/M24/M25) and leaving the index as a pure route table.

5. **The reference tree has holes where the code has surface.** `/v1/version` has no page (M21/M28). `fuse quickstart` has no page and no `meta.json` entry (M37). `Client.Version` is undocumented in the Go SDK reference (M39). Reference's contract is completeness; these are its most damaging kind of failure because a reader cannot know to look.

6. **`reference/` transcripts are unmaintained.** Every timestamped CLI transcript is fabricated, and several render tables in a style the CLI does not produce. This suggests transcripts were hand-written rather than captured. **The single highest-leverage structural change: capture CLI transcripts programmatically** (a golden-file test that renders each documented command against the stub and diffs the `.mdx` block) rather than authoring them. That converts an entire class of low-severity-but-numerous findings into a CI check.

7. **No page owns the single-replica constraint.** `reference/api/environments.mdx:206` links to a `cloud/scaling` page that does not exist (M38). `FleetManager` being single-process, single-replica by design is load-bearing operational knowledge with no home in `cloud/`.
