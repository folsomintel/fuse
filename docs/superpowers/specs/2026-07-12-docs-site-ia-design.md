# Fuse docs site — IA and content design

## Context

`docs/` already contains a Fumadocs (fumadocs-core/ui/mdx 16.10.7, Next.js 16) scaffold generated by `create-fumadocs`, with only placeholder content (`content/docs/index.mdx`, `content/docs/test.mdx`). The goal is to replace the placeholder with a real information architecture modeled on restate.dev's docs site: a tab-switchable nav (Learn / Docs) at the top of the sidebar, each tab with its own page tree.

Fuse itself: a self-hosted control plane for Firecracker microVMs (and optionally QEMU/GPU passthrough environments), driven by a CLI (`fuse`), three SDKs (Go/Python/TypeScript), and an HTTP API, with a declarative `Fusefile` spec format compiled client-side into orchestrator requests.

## Decisions from brainstorming

1. **Three top-level tabs**, not two: **Learn**, **Docs**, **Cloud / Self-host**. Fuse is self-hosted infra (not a library you `pip install`), so deployment/ops content is substantial enough to deserve its own tab rather than being folded into Docs.
2. **Learn tab** covers the full concept set: Fusefile, Providers (Firecracker vs QEMU), Hosts, Environments (VM lifecycle), Snapshots, Artifacts (rootfs bake / fc-agent / fused). No extra concept pages beyond this set (auth and GPU nuance are covered inline within Providers/relevant pages, not split out further).
3. **Docs tab** organized as three parallel sections — **SDKs**, **CLI**, **HTTP API** — rather than organized by resource-then-language. Each SDK section gets a getting-started + per-service reference; CLI gets a command reference; HTTP API gets an endpoint reference linking to the existing `internal/api/openapi.yaml`. Auth gets its own page in Docs since it cuts across all three surfaces.
4. **Cloud/Self-host tab** owns the full deployment lifecycle: orchestrator deploy (systemd/Postgres/TLS/env config), Firecracker host bring-up, GPU/QEMU host bring-up, and a dedicated **Troubleshooting** page.
5. **Known bug disclosure**: the fc-agent snapshot-restore NUL-corruption issue (confirmed 2026-07-12, see memory `fc-agent-restore-nul-corruption`) is documented publicly in Troubleshooting as a known limitation with a workaround (`sync` before snapshotting / restoring), not hidden.
6. **Build depth**: every page in the tree gets fully written content in this pass — nothing left as a stub. Pages backed by rich existing source (root `README.md`, `host-agent/README.md`, SDK READMEs, `openapi.yaml`, `.claude/AUTH.md`) get content adapted directly from that source. Pages needing net-new prose (Learn concept explanations, landing copy, troubleshooting narrative) get fully drafted, not sketched.

## Technical approach (verified against the installed Fumadocs version)

Confirmed via `docs/node_modules/fumadocs-core` and `fumadocs-ui` 16.10.7: there is **no `RootToggle`** (pre-v15 concept). The current mechanism is native and simple:

- One `defineDocs` collection, one `loader()`, one route (`app/docs/[[...slug]]/page.tsx`) — unchanged.
- Each top-level tab is a **folder under `content/docs/` with `meta.json` containing `"root": true`**. The page-tree builder promotes `root: true` folders into separate root nodes; `DocsLayout`'s `tabMode="top"` auto-derives the pill switcher from those roots via `getLayoutTabs()`.
- Nested groups within a tab (e.g. "CLI" as a collapsible group under Docs) are just subfolders with their own `meta.json`, no `root` flag.
- URLs will be prefixed `/docs/learn/...`, `/docs/docs/...`, `/docs/cloud/...` (the existing single `baseUrl: '/docs'` applies to all three). This is a URL-prefix cost accepted in exchange for not forking into three loaders/routes. Not worth the added complexity for this pass.

### File-level plan

- `docs/content/docs/learn/meta.json` — `{"title": "Learn", "root": true, "pages": [...]}`
- `docs/content/docs/reference/meta.json` — `{"title": "Docs", "root": true, "pages": [...]}` (folder named `reference/` on disk to avoid a confusing `content/docs/docs/` path; tab label is still "Docs")
- `docs/content/docs/cloud/meta.json` — `{"title": "Cloud", "root": true, "pages": [...]}`
- Existing `content/docs/index.mdx` and `content/docs/test.mdx` are removed (test.mdx) / repurposed (index.mdx becomes the top-level `/docs` redirect or a picker page — see Landing below).
- `docs/src/app/docs/layout.tsx` gains `tabMode="top"` on `DocsLayout`.
- `docs/src/lib/layout.shared.tsx` unchanged (nav title/github link only).
- `(home)/page.tsx` (the marketing landing page) gets real copy adapted from root `README.md`'s pitch, linking into `/docs/learn` and `/docs/docs`.

## Full page tree

### Learn tab (`content/docs/learn/`)

- `index.mdx` — **Overview**: what Fuse is, the one-paragraph pitch, a diagram-in-prose of control plane vs. hosts vs. VMs, links into Quickstart and each concept page.
- `quickstart.mdx` — **Quickstart**: install CLI, `fuse init`, `fuse connect`, `fuse up`, see it running — the fastest path to a running environment using the in-memory stub (no real host needed), then a pointer to Cloud/Self-host for a real Firecracker host.
- `concepts/fusefile.mdx` — **Fusefile**: the declarative spec format, full field reference (version, image, resources incl. gpu/gpu_kind, setup, services, run, workspace, expose, secrets), annotated version of the `fuse init` scaffold, how it compiles client-side into the orchestrator wire format (never sent as a Fusefile itself).
- `concepts/providers.mdx` — **Providers: Firecracker vs QEMU**: what a provider is (the `orchestrator.Provider` interface), Firecracker as the default/CPU backend, QEMU as the GPU-passthrough backend, stub mode for dev/tests, the explicit limitation that QEMU environments can't snapshot or fork (VFIO passthrough), how backend selection happens at host registration.
- `concepts/hosts.mdx` — **Hosts**: what a host is, registration (capacity, region, backend), cordon/uncordon, scheduling/fleet management, capacity vs. allocated.
- `concepts/environments.mdx` — **Environments**: the VM lifecycle state machine (provisioning → running → draining → destroying → destroyed/failed), create/get/list/destroy/drain/fork/rotate-token, live event streaming via SSE.
- `concepts/snapshots.mdx` — **Snapshots**: what a snapshot captures (disk-only, RAM field currently ignored), manual vs. auto mode, retention, restore semantics (in-place only), export refs, the GPU/QEMU exception (no snapshot support at all).
- `concepts/artifacts.mdx` — **Artifacts: rootfs bake, fc-agent, and fused**: the two distinct "agent" concepts disambiguated up front (fc-agent = host-side Python process driving Firecracker/QEMU; fused = reference in-guest Go daemon), what rootfs baking is and why the agent is baked into the image (`fc-bake-rootfs.sh`), what's baked in (podman-static, networking tooling, CA bundle, systemd unit), re-baking triggers, "bring your own agent" pointer to `host-agent/FUSE.md`'s contract.

### Docs tab (`content/docs/reference/`)

- `index.mdx` — **Overview**: map of SDKs vs CLI vs HTTP API, when to use which, links to auth.
- `auth.mdx` — **Authentication**: single shared-secret model (`ORCH_AUTH_TOKEN`), master token vs. revocable API keys, session cookie flow for browser/dashboard use (same-origin only), host-agent tokens (`FIRECRACKER_TOKEN`/`QEMU_AGENT_TOKEN`) as a separate concern, per-VM secrets model. Adapted from `.claude/AUTH.md`.
- `sdks/go.mdx` — **Go SDK**: install (`go get github.com/folsomintel/fuse/sdks/go`), `fuse.New(baseURL, token, opts...)`, `Client` fields, all four services' method signatures (Environments/Hosts/Snapshots/APIKeys), functional options, error handling.
- `sdks/python.mdx` — **Python SDK**: adapted from `sdks/python/README.md` — `uv add folsom-fuse`, context-manager `Client`, service methods, `ApiError`/predicates.
- `sdks/typescript.mdx` — **TypeScript SDK**: adapted from `sdks/typescript/README.md` — `bun add @folsom/fuse`, `FuseClient`, service methods (camelCase), `FuseApiError`/predicates, async-iterable events.
- `cli/index.mdx` — **CLI overview**: install, global flags (`--output`, `--context`, `--config`), config file location.
- `cli/contexts.mdx` — **Connecting & contexts**: `fuse connect`, `fuse context list/use/remove/current`.
- `cli/hosts.mdx` — **Hosts**: `fuse hosts`, `fuse host register/get/cordon/uncordon/remove/metrics`, full flag reference including GPU flags.
- `cli/environments.mdx` — **Environments**: `fuse environment list/get/create/destroy/drain/fork/rotate-token/watch`, `fuse up`, `fuse init`.
- `cli/snapshots.mdx` — **Snapshots**: `fuse snapshot create/list/get/delete/restore`.
- `cli/apikeys-metrics.mdx` — **API keys & metrics**: `fuse apikeys create/list/revoke` (master-token gated), `fuse metrics`.
- `api/index.mdx` — **HTTP API overview**: base URL, auth header, error envelope shape, link to full `openapi.yaml`.
- `api/environments.mdx` — endpoint reference: environments + snapshots routes, request/response shapes.
- `api/hosts-and-keys.mdx` — endpoint reference: hosts, api-keys, login/logout, health/ready/metrics.

### Cloud / Self-host tab (`content/docs/cloud/`)

- `index.mdx` — **Overview**: deployment model (self-hosted, single operator), what needs to run where (orchestrator + one or more hosts running fc-agent/qemu-agent).
- `orchestrator-deploy.mdx` — **Deploying the orchestrator**: build/run, full env var config table, Postgres vs in-memory state, TLS, `ORCH_REQUIRE_AUTH` strict mode, systemd co-location pattern adapted from root README's Deploy section.
- `firecracker-host-setup.mdx` — **Firecracker host setup**: hardware/KVM requirements (`ls -l /dev/kvm` check), one-command `bootstrap` vs. manual step-by-step, networking model (TAP/DNAT/iptables, ports to open), rootfs bake process walkthrough, adapted from `host-agent/README.md`.
- `gpu-host-setup.mdx` — **GPU (QEMU) host setup**: VFIO/IOMMU passthrough requirements, `qemu-vfio-bind.sh`, CUDA rootfs bake, `qemu-agent.service`, GPU E2E test, explicit callout that GPU environments can't snapshot/fork.
- `troubleshooting.mdx` — **Troubleshooting & known issues**: install/bake failures (missing `fused`/CA bundle in rootfs, netavark/nftables in bake environment), KVM permission issues, "stub mode" surprise (orchestrator silently runs in-memory if `FIRECRACKER_BASE_URL` unset), and the **known limitation**: snapshot restore can corrupt files to NUL bytes if the guest has unsynced writes at restore time — workaround is to `sync` in the guest before snapshotting/restoring; root cause is stale host page cache during the backing-file copy.

## Landing page

`(home)/page.tsx` gets real copy adapted from root `README.md`: headline pitch ("a control plane for agents — deploy Firecracker microVMs anywhere"), the 8-bullet feature list, two primary CTAs into `/docs/learn` (Learn) and `/docs/docs/cli` (Quickstart via CLI).

## Out of scope for this pass

- Renaming URLs to drop the `/docs/` prefix (would require forking into three loaders/routes — not worth it now).
- AI search / chat wiring beyond what's already scaffolded (`components/ai/search.tsx` stays as-is).
- Versioned docs (single-version site only).
- Dashboard/`fuse-frontend` product docs (out of this repo).
